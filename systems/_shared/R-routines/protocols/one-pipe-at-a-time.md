# One Pipe at a Time Protocol

## Purpose

Prevent cognitive overload, decision paralysis, and scattered energy by enforcing single-sprint focus. At any given moment, the Architect's work pipeline contains one active sprint for one project.

> From the Realization Manifesto, Rule 1: Ruthless Focus.

## Rules

1. **Single active sprint.** Only one project is in active build/analysis mode at any time. All other projects are in freeze until the current sprint completes.

2. **Sprint definition.** A sprint is a bounded piece of work: one deal analysis, one AI agent build, one financial model, one venture architecture. It has a clear deliverable and a clear "done" state.

3. **Freeze means freeze.** Frozen projects receive no creative energy, no "quick checks," and no background tinkering. The only exception is time-sensitive operational responses (e.g., a lead inquiry that must be answered today).

4. **Handoff triggers sprint end.** A sprint ends when one of these happens:
   - The deliverable is complete and handed to the Diplomat for execution
   - The deliverable is complete and the user explicitly moves to the next project
   - The user explicitly pauses or cancels the sprint

5. **Sprint tracking.** The active sprint should be visible in the system's `R-routines/state-map.md`. When checking state, identify:
   - What is the current active pipe?
   - What stage is it at? (analysis / build / review / handoff-ready)
   - What is the handoff criteria?

## When to Apply

- When the user starts discussing a new project while another sprint is active
- When multiple projects compete for attention in the same session
- When cognitive overload signals appear (scattered requests, frequent context-switching)

## How to Apply

1. **Acknowledge** the new idea or request
2. **Check** the current active sprint status
3. **Flag** the conflict: "You currently have [X] as your active sprint. Do you want to park the new idea and continue [X], or explicitly switch sprints?"
4. **Never silently juggle** two pipes — always surface the choice

## What This Is NOT

This protocol is NOT about refusing work. It is about surfacing the cost of context-switching and letting the user make an informed choice. The door stays open for urgent matters and genuine priority shifts.
